### 2. 決済システムの実装（バックエンド・フロントエンド）

#### 2.1 決済ゲートウェイの選択・導入
- [ ] 決済ゲートウェイの選定
  - 推奨: Stripe（日本対応、実装が容易）
  - その他: PayPal、Square、楽天ペイなど
- [ ] 決済ゲートウェイのアカウント作成
  - 本番環境用のAPIキーを取得
  - テスト環境用のAPIキーを取得（開発・テスト用）

#### 2.2 バックエンド実装
- [ ] 決済ゲートウェイSDKのインストール
  ```bash
  npm install stripe  # Stripeの場合
  ```
- [ ] 環境変数の設定
  ```env
  STRIPE_SECRET_KEY=sk_live_xxxxx  # 本番環境用
  STRIPE_PUBLISHABLE_KEY=pk_live_xxxxx  # 本番環境用
  STRIPE_WEBHOOK_SECRET=whsec_xxxxx  # Webhook用
  ```
- [ ] `src/services/paymentService.js`の修正
  - 現在のモック実装を実際の決済処理に置き換え
  - 決済ゲートウェイAPIとの連携実装
  - エラーハンドリングの実装
  - 返金処理の実装
- [ ] Webhookエンドポイントの実装
  - 決済完了通知の受信
  - 決済失敗通知の受信
  - 返金通知の受信
- [ ] 決済履歴の保存（既存の実装を確認・改善）

#### 2.3 フロントエンド実装
- [ ] 決済フォームの実装
  - カード情報入力フォーム（Stripe Elements使用推奨）
  - セキュリティ: カード情報は直接サーバーに送信せず、決済ゲートウェイに送信
- [ ] 決済処理のUI実装
  - ローディング表示
  - エラーハンドリング
  - 成功時のリダイレクト
- [ ] テスト決済の実施
  - テストカードでの決済テスト
  - エラーケースのテスト

#### 2.4 セキュリティ対策
- [ ] PCI DSS準拠の確認
  - カード情報を直接保存しない
  - 決済ゲートウェイのトークンを使用
- [ ] 決済情報の暗号化
- [ ] 不正アクセス対策

**現在の状態**: 決済処理はモック実装（実際の決済ゲートウェイ連携なし）  
**重要度**: ⚠️ **本番公開前に必須**

### 3. セキュリティ設定の最終確認（バックエンド）

#### 3.1 環境変数の確認
- [ ] `JWT_SECRET`が強力なランダム文字列（32文字以上）に設定されているか
- [ ] `DATABASE_URL`が本番環境用に設定されているか
- [ ] `CORS_ORIGIN`が本番環境のフロントエンドURLに設定されているか
- [ ] 機密情報が`.env`ファイルに含まれていないか（Gitにコミットされていないか）
- [ ] Renderの環境変数設定を確認

#### 3.2 セキュリティヘッダーの確認
- [ ] Helmet.jsの設定が適切か
- [ ] HTTPSが強制されているか
- [ ] `trust proxy`設定の修正（`app.set('trust proxy', 1)`に変更）
- [ ] Rate Limitingの設定確認

#### 3.3 データベースセキュリティ
- [ ] データベース接続がSSL/TLSで暗号化されているか
- [ ] データベースのバックアップが設定されているか
- [ ] データベースのアクセス制限が適切か

## 📝 注意事項

### 決済システムについて
- **現在の状態**: 決済処理はモック実装（実際の決済ゲートウェイ連携なし）
- **本番公開前**: 必ず実際の決済ゲートウェイ（Stripe、PayPalなど）を実装する必要がある
- **セキュリティ**: PCI DSS準拠を確認し、カード情報を直接保存しないこと

### 法的文書について
- 法的文書の内容は専門家（弁護士）に確認することを強く推奨
- 特に個人情報保護法、特定商取引法に準拠しているか確認

### セキュリティについて
- 本番環境では必ず強力なパスワードとJWT_SECRETを使用
- 機密情報は環境変数で管理し、Gitにコミットしない
- 定期的なセキュリティ監査を実施
