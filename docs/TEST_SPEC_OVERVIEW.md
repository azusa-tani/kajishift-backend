## 🔄 テスト実行フロー

### 推奨テスト実行順序

1. **準備フェーズ**
   - シードデータの投入確認
   - テスト環境へのアクセス確認
   - テスト用アカウントの確認

2. **管理者テスト**
   - 管理者としてログイン
   - 管理者機能の全テストを実行
   - テスト結果を記録

3. **依頼者テスト**
   - 依頼者としてログイン
   - 依頼者機能の全テストを実行
   - テスト結果を記録

4. **ワーカーテスト**
   - ワーカーとしてログイン
   - ワーカー機能の全テストを実行
   - テスト結果を記録

5. **統合テスト**
   - 複数ロール間の連携テスト
   - 承認プロセステスト
   - チャット機能の双方向テスト

## 📊 テスト結果記録方法

各テスト仕様書には「テスト結果記録」セクションがあります。以下の形式で記録してください：

| テストID | 実行日 | 結果 | 備考 |
|---------|--------|------|------|
| ADMIN-001 | 2026/02/27 | ✅ 合格 | |
| ADMIN-002 | 2026/02/27 | ❌ 不合格 | エラーメッセージ: ... |

### 結果の記録方法
- **✅ 合格**: 期待結果通りに動作した
- **❌ 不合格**: 期待結果と異なる動作、またはエラーが発生した
- **⚠️ 要確認**: 動作はするが、期待結果と一部異なる
- **⏸️ スキップ**: 前提条件が満たされず、テストを実行できなかった

## ⚠️ テスト実行前の確認事項

### 1. 環境確認
- [ ] フロントエンドが正常にデプロイされている
- [ ] バックエンドAPIが正常に動作している
- [ ] データベースにシードデータが投入されている

### 2. データ確認
- [ ] テスト用アカウントが存在する
- [ ] テスト用予約データが存在する（依頼者・ワーカーテスト用）
- [ ] テスト用メッセージデータが存在する（チャットテスト用）

### 3. 機能確認
- [ ] WebSocket接続が正常に動作している（チャット機能テスト用）
- [ ] ファイルアップロード機能が正常に動作している（画像添付テスト用）

## 🔍 テスト実行時の注意事項

### 1. テストデータの管理
- テスト実行前に、必要に応じてシードデータを再投入してください
- テストで作成したデータは、必要に応じて削除してください
- 本番環境でテストする場合は、テストデータの削除を忘れずに行ってください

### 2. テストの独立性
- 各テストは独立して実行できるように設計されていますが、一部のテストは順序に依存する場合があります
- 順序に依存するテストは、テスト仕様書に明記されています

### 3. エラーハンドリング
- エラーが発生した場合は、エラーメッセージとスクリーンショットを記録してください
- ブラウザのコンソールログも確認してください

### 4. パフォーマンス
- テスト実行中にパフォーマンスの問題を発見した場合は、記録してください
- 特に、ページの読み込み時間、APIの応答時間に注意してください

## 📝 テスト結果の集計

テスト実行後、以下の情報を集計してください：

1. **合格率**: 合格したテスト数 / 全テスト数 × 100
2. **不合格テスト一覧**: 不合格となったテストのIDと原因
3. **要確認テスト一覧**: 要確認となったテストのIDと詳細
4. **発見されたバグ一覧**: バグの詳細、再現手順、優先度
